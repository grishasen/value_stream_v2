    Value Stream — Implementation Kick-off Prompt¶

    Paste this entire document into a fresh agent session (or hand it to a new engineer) to start building Value Stream. It is self-contained: every reference is a path, every deliverable has acceptance criteria, every constraint is explicit.

    0. Who you are and what you are doing¶

    You are an engineer (or coding agent) tasked with building a new application called Value Stream. Value Stream is a configuration-driven, aggregate-first business intelligence platform — it ingests file-based exports, reduces them to small mergeable sufficient statistics during one chunk pass, and serves business reports and dashboards from those persisted aggregates. Raw event rows are never stored.

    The full specification already exists. Do not invent design choices. When in doubt, read the spec, then act. When the spec is silent, ask before deciding.

    You will be working in a separate folder dedicated to Value Stream. The spec docs live somewhere else (see §2) and stay read-only. Your code goes in the new folder.

    This prompt covers Phase 0 only (Foundations, ~1–2 weeks). After Phase 0 ships, follow the phased plan in design/implementation-plan.md for what's next.

    1. Target folder¶

    Create a new folder for the project. Default suggestion (adjust if your environment differs):

    TARGET_DIR = /sessions/cool-practical-fermat/mnt/proof_of_value/valuestream

    If this path is not writeable in your environment, ask the operator for an alternative path before creating files. Do not pollute the spec folder at /sessions/cool-practical-fermat/mnt/proof_of_value/.

    Initialize an empty git repo at TARGET_DIR and commit your work as you go. First commit message: chore: scaffold repository.

    2. Read these specs first¶

    The complete specification for Value Stream is in /sessions/cool-practical-fermat/mnt/proof_of_value/. Read in this order before writing any code:

    1. docs/meta/documentation-guide.md — reading order and a doc-completeness checklist.
    2. docs/concepts/architecture.md — high-level architecture, NFRs, deployment.
    3. docs/concepts/domain-model.md — glossary; memorize the vocabulary before reading further.
    4. docs/design/replacement-design.md — master design with storage layout, YAML DSL, APIs, migration.
    5. docs/design/implementation-plan.md — repo layout, dependency pins, testing strategy, Phase 0 §5.
    6. docs/reference/expression-dsl.md — the AST grammar you will be implementing in Phase 0.
    7. docs/reference/processors.md — context for what the AST will eventually feed (you will not implement processors in Phase 0).
    8. docs/reference/readers-and-formats.md — context only.
    9. docs/reference/algorithms.md — context only.
    10. docs/reference/chart-catalog.md — context only.
    11. docs/reference/faq.md — keep open; refer when stuck.

    The visual mockups in docs/mockups/ are useful for orientation but not required reading for Phase 0.

    You may not modify any file under /sessions/cool-practical-fermat/mnt/proof_of_value/ unless the operator explicitly asks. Treat it as read-only.

    3. Phase 0 scope — what you will build¶

    Phase 0 lays the foundations: the repo, the tooling, the configuration loader, the expression DSL, and the metadata schema. You will not build any processor, reader, or chart yet. Resist scope creep.

    Concretely, by the end of Phase 0 the repo must contain:

    3.2 Tooling and dependencies¶

    Use uv for environment management. The dependency block in pyproject.toml must match design/implementation-plan.md §3. For Phase 0 you only need a subset; install the others lazily later. Phase 0 minimal dependencies:

    [project]
name = "valuestream"
version = "0.1.0"
requires-python = ">=3.11"
dependencies = [
    "polars>=1.16,<2",
    "duckdb>=1.1,<2",
    "pyarrow>=18,<19",
    "pydantic>=2.7,<3",
    "jsonschema>=4.22,<5",
    "pyyaml>=6.0,<7",
    "click>=8.1,<9",
    "rich>=13.7,<14",
    "structlog>=24.4,<25",
]
[project.optional-dependencies]
dev = [
    "pytest>=8.3", "pytest-asyncio>=0.24",
    "hypothesis>=6.112", "ruff>=0.7", "mypy>=1.13",
    "mkdocs>=1.6", "mkdocs-material>=9.5",
    "pre-commit>=4.0",
]

    Add datasketches, scipy, scikit-learn, polars-ds, fastapi, uvicorn, streamlit, plotly, etc. only when Phase 1+ requires them.

    3.3 The expression DSL — the centerpiece of Phase 0¶

    Implement the closed AST grammar in docs/reference/expression-dsl.md end-to-end. Required modules:

    • valuestream.expr.ast — Pydantic v2 model classes for every node kind (atom, unary, nary, predicate, conditional, datetime). Use a discriminated union on the op field.
    • valuestream.expr.parser — turn a YAML/JSON dict into an AST instance. Tolerant of YAML idiosyncrasies (e.g. {op: in, column: c, values: [...]} is valid).
    • valuestream.expr.validator — type-check the AST against a column-name → dtype map (provided by the config). Implements docs/reference/expression-dsl.md §3` (type rules) and §6 (validation rules). Errors carry the AST path.
    • valuestream.expr.translator — turn an AST into a polars.Expr. Implements every row in docs/reference/expression-dsl.md §4(mapping table). One Polars expression per node — nevereval`.

    JSON Schema for the AST goes at schemas/expr.json (Draft 2020-12).

    3.4 Configuration loader and canonicalization¶

    • valuestream.config.model — Pydantic models for Catalog, Pipeline, Source, Reader, Transform, Processor, Metric, Dashboard, Tile, Dimension. Map exactly to the YAML shapes in design/replacement-design.md §7 and docs/reference/processors.md.
    • valuestream.config.loader — read the five YAML files from a catalog directory, validate each against its JSON Schema, instantiate the typed model. Returns a Catalog object.
    • valuestream.config.canonical — produce a canonical YAML body for hashing per docs/reference/expression-dsl.md §7. Output sha256 hex string. **Two YAMLs that parse to the same AST must produce the sameconfig_hash`.**

    3.5 Metadata schemas¶

    • valuestream.store.meta — DDL for meta/chunks.duckdb, meta/pipeline_runs.duckdb, meta/config_versions.duckdb, meta/lineage.duckdb. Schemas exactly as in design/replacement-design.md §6.4 and docs/concepts/architecture.md §6.1. Provideinit_meta_dbs(workspace_path)` that creates the four DBs idempotently. No data writes yet — just the schemas.

    3.6 CLI — the only Phase 0 entry point¶

    Implement one CLI command using Click:

    valuestream validate <workspace_dir>

    It loads the catalog from <workspace_dir>/catalog/, validates each YAML against its JSON Schema, type-checks every expression in transforms / processors / metrics, and prints structured success or errors. Exit code 0 on success, 1 on failure.

    Other CLI commands (run, vacuum, serve, migrate, backfill, probe) come in later phases. Stub them as NotImplementedError if you want, but do not implement them.

    3.7 Example workspace¶

    Hand-write examples/workspace_demo/catalog/ so it covers every operator in reference/expression-dsl.md and every processor kind in reference/processors.md (configs only — no actual data needed). Use this as a smoke-test target for valuestream validate.

    3.8 Tests¶

    Required unit tests for Phase 0:

    • tests/unit/test_expr_parser.py — round-trip parse for every operator in reference/expression-dsl.md §2.
    • tests/unit/test_expr_validator.py — happy path + at least one failing case per error type in §6.
    • tests/unit/test_expr_translator.py — for each AST node kind, build a tiny Polars LazyFrame, run the translated expression, assert the result.
    • tests/unit/test_config_loader.py — load examples/workspace_demo/catalog/; assert the typed Catalog matches expectations.
    • tests/unit/test_config_canonical.py — two YAMLs that differ only in key order / whitespace produce the same config_hash. A meaningful change produces a different hash.
    • tests/unit/test_store_meta.py — init_meta_dbs creates all four DBs with the expected tables; idempotent on re-run.
    • tests/property/test_expr_roundtrip.py — Hypothesis property test: for any randomly generated AST that passes the validator, parsing → canonicalizing → re-parsing yields an equivalent AST.

    Coverage gate: ≥ 90% on valuestream.expr and valuestream.config.

    3.9 Local quality gates¶

    This repository intentionally has no checked-in GitHub Actions workflow. Run local gates before hand-off:

    1. uv sync --all-extras
    2. uv run ruff check .
    3. uv run ruff format --check .
    4. uv run mypy src
    5. uv run pytest -m "not bench and not slow"
    6. uv run valuestream validate examples/workspace_demo
    7. uv run mkdocs build --strict

    3.10 Docs site¶

    mkdocs.yml building the docs from docs/ (the spec folder, mounted read-only or symlinked into the Value Stream repo's docs/ directory). The local mkdocs serve and mkdocs build --strict commands must work.

    4. Coding standards¶

    • Python 3.11+, type hints everywhere, no from __future__ import clutter.
    • mypy --strict for valuestream.config, valuestream.expr, valuestream.store. Other modules: --check-untyped-defs.
    • Ruff format before committing; the pre-commit hook enforces this.
    • No eval, no exec, no getattr-by-string, no dynamic imports. The whole point of the AST is to avoid these.
    • No silent failures. Every error path raises a typed exception and is tested.
    • Docstrings on every public function and class. Use Google style.
    • Sentence case in all user-facing strings (CLI messages, errors, doc strings, headings). No Title Case, no ALL CAPS.
    • Logs are structured JSON via structlog. pipeline_run_id and chunk_id are reserved keys.
    • Commits follow Conventional Commits (feat:, fix:, chore:, test:, docs:, refactor:).

    5. Quality gates and Definition of Done for Phase 0¶

    Phase 0 is done when all of these pass:

    1. uv sync succeeds; valuestream --help prints the CLI banner.
    2. ruff check . and ruff format --check . are clean.
    3. mypy src/ is clean.
    4. pytest -m "not bench and not slow" is green; coverage ≥ 90% on valuestream.expr and valuestream.config.
    5. valuestream validate examples/workspace_demo exits 0.
    6. The Hypothesis property test in test_expr_roundtrip.py runs ≥ 200 examples without flakiness.
    7. Local quality gates pass.
    8. mkdocs serve renders the docs site without errors.
    9. The repo has at least 5 commits with conventional messages and a clear history.
    10. A short README.md at the repo root explains: what Value Stream is, how to set up the env, how to run the validator, where the spec lives.

    When all 10 are true, tag the commit v0.1.0-phase0 and stop. Do not start Phase 1 without operator confirmation.

    6. Explicit non-goals for Phase 0¶

    Do not do any of these in Phase 0. They are Phase 1+:

    • Any reader (pega_ds_export, parquet, csv, xlsx).
    • Any transform (rename_capitalize, parse_datetime, derive_calendar, derive_action_id, filter, dedup, defaults, cast, drop_columns, coalesce).
    • Any processor (binary_outcome, numeric_distribution, score_distribution, entity_lifecycle, entity_set, funnel, snapshot).
    • Any state type beyond schema declarations (no count, tdigest, hll, etc. computation).
    • Any chart kind, Plotly figure, or Streamlit page.
    • Any FastAPI route or MCP server.
    • The valuestream run, valuestream vacuum, valuestream migrate, valuestream backfill, or valuestream serve CLI commands.
    • The legacy → Value Stream migration tool.

    If you find yourself reaching for any of these, stop and re-read this section.

    7. Suggested day-by-day order¶

    A reasonable ten working days for Phase 0 (one engineer, full-time). Adjust as you wish, but do not skip steps.

    Day Focus
    1 Repo skeleton, pyproject.toml, uv sync, ruff/mypy/pre-commit baseline, first commit.
    2 valuestream.expr.ast types; schemas/expr.json first draft; round-trip parse for atoms + comparisons.
    3 valuestream.expr.parser for arithmetic + nary + conditional + datetime nodes.
    4 valuestream.expr.validator (type + column-existence + per-rule from §6).
    5 valuestream.expr.translator covering every entry in §4 mapping table.
    6 valuestream.config.model + valuestream.config.loader + per-file JSON Schemas.
    7 valuestream.config.canonical + hashing; two-different-YAMLs-same-hash test.
    8 valuestream.store.meta DuckDB DDL; init_meta_dbs + idempotency tests.
    9 valuestream validate CLI; example workspace; integration test.
    10 Property test (Hypothesis) for AST round-trip; local gate run; tag v0.1.0-phase0.

    8. When to ask for help¶

    Stop and ask the operator before:

    • Choosing a file path different from this prompt's defaults.
    • Adding a dependency not in §3.2 or design/implementation-plan.md §3.
    • Diverging from any algorithm, schema, or DSL grammar in the spec.
    • Inventing a new operator, state type, or processor kind.
    • Modifying any file under /sessions/cool-practical-fermat/mnt/proof_of_value/.

    If the spec is silent on a question, call it out — propose two options with trade-offs, and let the operator choose. Do not silently default.

    9. Operator hand-off checklist¶

    When you believe Phase 0 is complete, post a single message to the operator with:

    1. The path to the new repo.
    2. The output of valuestream validate examples/workspace_demo.
    3. The output of pytest --cov=valuestream.
    4. The local quality-gate command outputs.
    5. A two-paragraph summary of any decisions you made where the spec was silent.
    6. The git log (one line per commit).

    Do not start Phase 1 work; wait for explicit confirmation.

    11. Begin¶

    Confirm you have read this prompt. Then:

    1. Create TARGET_DIR (or ask for an alternative path).
    2. Read the four required spec docs (INDEX, ARCHITECTURE, DOMAIN_MODEL, IMPLEMENTATION_PLAN), then EXPRESSION_DSL in depth.
    3. Start with day 1: repo skeleton + pyproject.toml + tooling. Commit with message chore: scaffold repository.
    4. Proceed through the day-by-day order in §7 unless you have a strong reason to deviate.

    When in doubt: read the spec, then act. When the spec is silent: ask.
